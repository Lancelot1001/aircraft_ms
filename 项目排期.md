## 阶段二：核心业务逻辑——部件管理（入库、安装、拆卸、更换）

**目标**：实现部件全生命周期中"入库→安装→拆卸→更换"这一主链路的正确业务逻辑，完成事务场景一（部件更换事务）。

### 任务清单

| # | 任务 | 说明 |
|---|------|------|
| 2.1 | Express 项目骨架搭建 | 初始化 npm 项目，安装 express / mysql2 / ejs 依赖，建立目录结构 |
| 2.2 | 数据库连接层 | 封装 MySQL 连接池，统一错误处理 |
| 2.3 | 部件入库功能 | 录入新部件，自动验证重复编号 |
| 2.4 | 部件安装功能 | 检查部件可用性（未退役、未安装在其他飞机），插入安装记录（end_time=NULL 表示当前有效） |
| 2.5 | 部件拆卸功能 | 关闭安装记录（设置 end_time），保留历史不删除 |
| 2.6 | 部件更换事务 | **事务场景一**：关闭旧记录→检查新部件→插入新记录→更新状态，全部在一个事务中 |
| 2.7 | 安装唯一性数据库层保障 | 触发器 + 部分唯一索引，确保同一部件同一时刻只有一条有效安装记录 |

### 关键业务规则落地

| 规则 | 实现方式 |
|------|----------|
| 安装唯一性 | 数据库层：INSERT 前触发器检查 + 唯一约束；应用层：Service 层预检 |
| 历史保留 | InstallationRecord 使用 `installed_at` + `removed_at` 时间区间，拆卸只设 `removed_at`，从不 DELETE |
| 退役不可安装 | 数据库触发器：BEFORE INSERT ON InstallationRecord 检查 Component.status ≠ 'retired' |
| 时间合理性 | CHECK 约束：`removed_at IS NULL OR removed_at > installed_at` |

### 事务场景一：部件更换

```
BEGIN TRANSACTION
  1. UPDATE InstallationRecord SET removed_at = NOW() 
     WHERE component_id = ? AND removed_at IS NULL    -- 关闭旧记录
  2. SELECT status FROM Component WHERE id = ?          -- 检查新部件
     → IF status = 'retired' → ROLLBACK
  3. SELECT COUNT(*) FROM InstallationRecord 
     WHERE component_id = ? AND removed_at IS NULL      -- 检查是否已安装
     → IF COUNT > 0 → ROLLBACK
  4. INSERT INTO InstallationRecord (...)               -- 插入新记录
  5. COMMIT
```

### 验收标准

- [ ] `npm install && npm start` 可正常启动服务
- [ ] 部件入库：输入部件编号、型号、批次，成功存入，重复编号被拒绝
- [ ] 部件安装：选择可用部件→选择飞机→输入安装位置，成功创建安装记录（end_time=NULL）
- [ ] 部件拆卸：选择已安装部件→输入拆卸原因，成功关闭安装记录（end_time=当前时间），历史记录保留
- [ ] 部件更换事务：一步操作完成"拆旧装新"，中间任何一步失败则全部回滚
- [ ] 非法拦截（数据库层）：已安装在其他飞机的部件再次安装时被拒绝

## 阶段三：核心业务逻辑——维修管理与退役处理

**目标**：实现维修登记、维修完成、部件退役功能，完成事务场景二（退役处理事务）和事务场景三（维修完成事务）。

### 任务清单

| # | 任务 | 说明 |
|---|------|------|
| 3.1 | 维修登记功能 | 为部件创建维修工单，记录维修类型、开始时间、责任人员 |
| 3.2 | 维修完成功能 | 更新维修结果、结束时间，必要时调整部件状态为 available |
| 3.3 | 维修完成事务 | **事务场景三**：记录维修结果→更新结束时间→调整部件状态 |
| 3.4 | 部件退役功能 | 检查部件状态→若仍安装则先拆卸→写入退役记录→更新部件状态为 retired |
| 3.5 | 退役处理事务 | **事务场景二**：检查安装状态→(若需则关闭安装记录)→写入退役记录→标记 retired |
| 3.6 | 操作人员关联 | 安装、拆卸、维修、退役操作均关联 Operator 表，记录责任人 |

### 业务规则落实（宽松模式说明）

- **维修前提（宽松模式）**：部件状态为 `installed` 或 `available`（拆卸后送修）均可创建维修记录
- **维修完成后**：部件状态更新为 `available`（维修完成可重新安装）
- **退役前提**：若部件当前处于 `installed` 状态，必须先拆卸（关闭安装记录）再退役
- **退役后**：部件标记 `retired`，所有后续安装操作被数据库层拒绝

### 事务场景二：退役处理

```
BEGIN TRANSACTION
  1. SELECT status FROM Component WHERE id = ? 
     → IF status = 'retired' → ROLLBACK
  2. IF status = 'installed':
     UPDATE InstallationRecord SET removed_at = NOW() 
     WHERE component_id = ? AND removed_at IS NULL
  3. INSERT INTO ScrapOrRetirementRecord 
     (component_id, reason, retired_at, approved_by)
  4. UPDATE Component SET status = 'retired' WHERE id = ?
  5. COMMIT
```

### 事务场景三：维修完成

```
BEGIN TRANSACTION
  1. UPDATE MaintenanceRecord 
     SET maintenance_result = ?, completed_at = NOW() 
     WHERE id = ? AND completed_at IS NULL
  2. IF maintenance_result = '修复完成':
     UPDATE Component SET status = 'available' WHERE id = ?
  3. COMMIT
```

### 验收标准

- [ ] 维修登记：选择部件→输入维修类型→关联操作人员，成功创建维修记录
- [ ] 维修完成：更新维修结果，部件状态自动调整为 `available`
- [ ] 维修完成事务：若更新失败则维修记录状态不变，部件状态不变
- [ ] 退役处理：部件标记为 retired，退役记录写入成功
- [ ] 退役后部件不可再安装（数据库触发器拦截）
- [ ] 仍处于安装状态的部件退役时，自动先关闭安装记录
- [ ] 所有操作均关联具体操作人员

## 阶段四：飞行日志与部件使用关联分析

**目标**：实现飞行日志登记功能，建立飞行记录与部件安装记录的关联查询，支持部件使用情况统计分析。

### 任务清单

| # | 任务 | 说明 |
|---|------|------|
| 4.1 | 飞行日志登记 | 记录飞机飞行任务：起飞/降落时间、飞行时长、任务类型 |
| 4.2 | 飞行日志时间约束 | CHECK 约束确保飞行时间区间合理，飞行时长与起降时间一致 |
| 4.3 | 飞行-部件关联查询 | 给定时间段，查询该时间段内某飞机上安装了哪些部件（利用安装记录的时间区间与飞行时间区间做交集判断） |
| 4.4 | 部件飞行统计 | 统计某部件在其安装期间经历的飞行次数、累计飞行时长 |

### 核心查询逻辑

飞行与部件的关联基于**时间区间交集**：

```
某飞行记录与某部件安装记录相关 ⇔ 
  flight.start_time < installation.removed_at (或 IS NULL)
  AND flight.end_time > installation.installed_at
```

### 验收标准

- [ ] 飞行日志：可录入飞行任务，系统校验起降时间合理性
- [ ] 关联查询：给定飞机+时间段，可查出该时段内安装在飞机上的所有部件
- [ ] 部件统计：给定部件编号，可统计该部件历史上经历的飞行次数和累计飞行时长
- [ ] 时间合理性：起飞时间 < 降落时间，飞行时长与差值一致

```
         阶段六（Web前端）
              │
              ▼
         阶段七（集成测试与演示验证）
              │
              ▼
         阶段八（文档与交付）
```

- 阶段二和三可部分并行（维修/退役的数据库层可与部件管理共用）
- 阶段四与阶段三无严格依赖，可并行
- 阶段五依赖二、三、四的数据和业务逻辑就绪
- 阶段六依赖一到五的核心功能完成
- 阶段七必须在阶段六之后
- 阶段八可在阶段七后半段开始

```
├── approval_info             — 审批信息
└── notes

Operator

├── operator_code (UNIQUE)    — 工号

└── notes
```
